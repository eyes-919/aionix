# Aionix 開発に必要なエージェント一覧

作成日: 2026-04-25

## 1. この文書の目的

この文書は、Aionix フレームワークそのものを開発・運用するために必要なエージェント一覧を定義する。

Aionix 設計では Agent と Skill を明確に分離し、導入単位を Package とする。本文書では、Aionix の lifecycle を通じて、どのような判断主体（Agent）が必要か、それぞれの役割と責務を整理する。

## 2. Aionix 開発ライフサイクルのフェーズ

Aionix を継続的に開発・運用するプロセスは、以下の段階を想定する。

```
設計・計画 → 実装 → テスト・検証 → ドキュメント → レビュー → リリース・デプロイ
   Phase 1    Phase 2      Phase 3        Phase 4      Phase 5        Phase 6
```

各フェーズで必要なエージェント（判断主体）を定義する。

## 3. エージェント定義フレームワーク

各エージェントは以下を定義する。

```yaml
name: エージェント名
role: 主な役割
goal: 達成する目的
phase: ライフサイクルフェーズ
decision_scope: 判断対象
primary_skills:
  - Skill 1
  - Skill 2
permissions:
  read: [対象]
  write: [対象]
  approval_required: [操作]
```

## 4. エージェント一覧

### Phase 1: 設計・計画フェーズ

#### 4.1.1 Architecture Designer Agent

**役割:** Aionix 全体のアーキテクチャ仕様を設計・更新する判断主体

**目的:**
- システム設計書の作成・改定
- 新しい機能領域の設計
- エージェント、スキル、パッケージの構成設計
- 技術選定、トレードオフ分析

**判断スコープ:**
- 全体アーキテクチャの妥当性
- 新機能の位置づけ
- Phase間の依存関係
- 既存設計との整合性

**主要 Skill:**
- Markdown Formatter Skill
- Diagram Generator Skill
- Specification Validation Skill
- Cross-reference Resolver Skill

**権限:**
- 読取: 既存設計書、GitHub Issue、PR
- 書込: design/ 配下の .md ファイル、design.yaml
- 削除: 承認が必要

**特記:**
- 人間レビューを必須とする
- Major version change には Steering Committee 承認を必須

---

#### 4.1.2 Planning Agent

**役割:** 各フェーズのタスク分解と実装計画を立案する

**目的:**
- 大型フィーチャーを実装可能な単位に分解
- タスク間の依存関係を整理
- リソース見積もり
- スケジュール調整

**判断スコープ:**
- タスク粒度の適切性
- 並列実行可能なタスク
- クリティカルパス
- リスク評価

**主要 Skill:**
- Task Decomposition Skill
- Dependency Analysis Skill
- Work Breakdown Structure Skill
- Risk Assessment Skill

**権限:**
- 読取: 設計書、Issue テンプレート、既存 Issue
- 書込: GitHub Issue 作成・更新、Project board
- 削除: 承認が必要

**特記:**
- GitHub Project board 連携
- 自動的に Issue チェックリストを生成

## 5. 横断的・共通エージェント

上記のフェーズ別エージェント以外に、複数フェーズで必要な共通エージェントがある。

### 5.1 File Governor Agent

**役割:** ファイル操作時の保存場所・命名規則判断

**目的:**
- ルート直下への不正な成果物保存を防止
- LHF に従ったディレクトリ配置を強制
- ファイル命名規則を適用
- 既存ファイルか新規作成かを判断

**判断スコープ:**
- 保存先の妥当性
- ファイル名の命名規則適合性
- 更新 vs 新規の判定
- 削除操作時の保護

**主要 Skill:**
- Path Validator Skill
- File Naming Rule Enforcer Skill
- Duplicate Detector Skill

**権限:**
- 読取: ディレクトリ構造、既存ファイル
- 拒否権: ルート直下への書込、LHF 違反ファイル保存、不適切な削除

**特記:**
- 全エージェントが呼び出す共通ゲートキーパー
- 他エージェントが File 操作時は必ず File Governor を経由

---

### 5.2 Context Manager Agent

**役割:** エージェント実行時のコンテキスト整理・提供

**目的:**
- 会話履歴・依頼内容の要約
- 関連ドキュメントの抽出・提供
- 既存ルール・ポリシーの提供
- 制約条件の明示

**判断スコープ:**
- 提供すべき情報の選別
- 情報の優先度付け
- 関連性の判定
- 秘密情報の除外判定

**主要 Skill:**
- Context Summarizer Skill
- Document Relevance Finder Skill
- Rule Extractor Skill
- Constraint Analyzer Skill

**権限:**
- 読取: すべてのドキュメント、設定、Issue、PR

---

### 5.3 Approval Gate Agent

**役割:** 高リスク操作の人間承認ゲートキーパー

**目的:**
- 削除操作の承認
- 本番環境操作の承認
- 破壊的変更（breaking change）の承認
- セキュリティ関連操作の承認

**判断スコープ:**
- 操作リスク度の判定
- 承認者の選別
- 承認タイムアウト管理

**主要 Skill:**
- Risk Classifier Skill
- Approval Requester Skill
- Timeout Manager Skill

**権限:**
- 書込: 承認要求（Issue、PR comment）
- 読取: 承認状態

---

## 6. 初期実装の優先度

すべてのエージェントを同時に実装することは現実的ではない。推奨される優先度（MVP → Phase 1 → Phase 2）は以下の通り。

### MVP（最小実行可能エージェント構成）

必須:
- [ ] Architecture Designer Agent
- [ ] Planning Agent
- [ ] Core Implementation Agent
- [ ] Code Reviewer Agent
- [ ] File Governor Agent
- [ ] Context Manager Agent

オプション但し強く推奨:
- [ ] Test Design Agent
- [ ] Release Manager Agent

### Phase 1（初期リリース後の拡張）

- [ ] Agent Layer Implementation Agent
- [ ] Skill Implementation Agent
- [ ] Test Executor Agent
- [ ] Technical Documentation Agent
- [ ] Quality Gate Agent
- [ ] Package Manager Agent

### Phase 2（二次機能）

- [ ] Performance Validator Agent
- [ ] User Documentation Agent
- [ ] Release Notes Agent
- [ ] Architecture Reviewer Agent
- [ ] Documentation Reviewer Agent
- [ ] Deployment Agent
- [ ] Observability Agent

---

## 7. 次のステップ

本文書で定義したエージェント一覧に基づいて、以下を実施する。

1. **Agent Definition Specification** - 各エージェントのメタデータ仕様を定義
2. **Skill Registry Design** - 各エージェントが依存する Skill の詳細設計
3. **Package Manifest Template** - Agent Package の manifest テンプレート作成
4. **Permission Policy Definition** - 各エージェントの権限制御ポリシーを詳細化
5. **Agent Interaction Protocol** - エージェント間の通信・連携プロトコルを定義
