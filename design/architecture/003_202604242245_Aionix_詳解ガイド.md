# Aionix 詳解ガイド

作成日: 2026-04-24

## 1. この文書の位置づけ

この文書は、`Aionix 全体設計` を人間が理解し、実装・運用・公開リポジトリ化へ進めるための詳しい解説書である。

`Aionix 全体設計` がアーキテクチャの骨格を定義する文書だとすれば、この文書は、その骨格をどのように理解し、どの順序で形にし、どのような判断基準で拡張していくかを説明する実践ガイドである。

Aionix は、AI を便利なチャット相手として扱うのではなく、AI エージェントが継続的に作業し、成果物を作り、レビューされ、改善されるための実行基盤として捉える。

そのため、この文書では単なる用語説明ではなく、次の観点を重視する。

- AI 実行環境を OS 的に捉える理由
- Agent と Skill を分ける理由
- Claude Code / CLI / MCP / Hooks / Skills をどう位置づけるか
- ファイル、タスク、ログ、権限、レビューをどう管理するか
- 最小実装から公開可能なフレームワークへどう進めるか

## 2. Aionix が解こうとしている問題

現在の AI 活用は、多くの場合、個別のプロンプト、個別のチャット、個別のツール利用に閉じている。

人間は毎回、次のようなことを手作業で判断している。

- どのファイルを読ませるべきか
- どのルールを守らせるべきか
- どこに成果物を保存させるべきか
- 既存ファイルを更新すべきか、新規作成すべきか
- どのツールを使わせるべきか
- 失敗したときに何を確認すべきか
- どこまで AI に任せ、どこから人間が承認すべきか

この状態では、AI が高度化するほど、人間側の運用負荷も増える。

AI にできることが増える一方で、作業の再現性、成果物の所在、判断ログ、権限管理、レビュー責任が曖昧になりやすい。

Aionix は、この問題に対して、AI 実行環境を OS 的な構造で整理する。

つまり、AI に作業を頼むたびに場当たり的に指示するのではなく、次のような構造を先に定義する。

- Agent は何を判断するのか
- Skill は何を実行するのか
- Context はどこから来るのか
- Memory はどの範囲で使えるのか
- Artifact はどこへ保存するのか
- Permission は誰が持つのか
- Log は何を残すのか
- Human approval はどこで必要か

## 3. OS 的に考える理由

OS は、計算機資源を直接人間やアプリケーションに触らせず、抽象化して管理する。

Linux であれば、プロセス、ファイル、ソケット、ユーザー、権限、シグナル、ログ、パッケージ、デバイスといった概念を提供する。

Aionix も同じように、AI 作業に必要な概念を抽象化する。

AI に対して毎回「このファイルを読んで、こう判断して、ここへ保存して」と言うのではなく、作業の土台として以下を持たせる。

- Agent Runtime
- Skill Registry
- Context Manager
- Permission Manager
- Artifact Manager
- Task Manager
- Observation Logger
- Policy Engine

これにより、AI 作業は単発の会話ではなく、管理可能な実行単位になる。

## 4. OS概念との対応を詳しく見る

### 4.1 Process = Agent 実行インスタンス

OS における Process は、実行中のプログラムである。

Aionix では、Agent の一回の実行が Process に相当する。

たとえば `Document Writer Agent` が「Aionix の解説書を作る」というタスクを受け取って動く場合、その実行単位が Process 的に扱われる。

Process として扱うことで、以下が可能になる。

- 開始時刻を記録する
- 入力を記録する
- 利用した Skill を記録する
- 出力した Artifact を記録する
- 成功・失敗を記録する
- 中断や再実行の対象にできる

### 4.2 Thread = Agent 内のサブタスク

一つの Agent 実行の中でも、複数の小さな作業がある。

たとえば文書作成では、以下のようなサブタスクがある。

- 既存文書を読む
- 構成を決める
- 本文を書く
- Markdown を整える
- 保存先を決める
- GitHub に保存する

これらは Thread 的に扱える。

初期実装では実際の並列実行までは不要だが、将来的にはサブタスク単位でログを残し、失敗時にどこからやり直すかを判断できるようにする。

### 4.3 System Call = Tool / Skill 呼び出し

OS では、アプリケーションがファイルを開く、ネットワークに接続する、プロセスを作るといった操作を System Call 経由で行う。

Aionix では、Agent が外部操作を行うときに Tool や Skill を呼び出す。

例:

- GitHub にファイルを作成する
- Markdown を整形する
- Excel を生成する
- PDF を出力する
- Web から情報を取得する
- ローカルスクリプトを実行する

この呼び出しを System Call 的に扱うことで、権限チェック、ログ記録、失敗処理を一貫させられる。

### 4.4 Library = Skill

Skill はライブラリである。

ここが Aionix の重要な設計ポイントである。

Skill は判断主体ではない。Skill は、Agent から呼び出される共通機能である。

たとえば `Markdown Formatter Skill` は、Markdown の整形を担当する。だが、どの文書を整形すべきか、整形後に保存すべきか、レビューを要求すべきかは判断しない。

それを判断するのは Agent である。

この分離により、同じ Skill を複数の Agent が使える。

例:

- Document Writer Agent が Markdown Skill を使う
- Reviewer Agent が Markdown Skill を使う
- Release Manager Agent が Markdown Skill を使う
- Training Material Agent が Markdown Skill を使う

Skill をライブラリとして扱うことで、Aionix は拡張しやすくなる。

### 4.5 File System = Workspace / Repository / Artifact Store

AI が生成する成果物は、どこかに保存されなければならない。

チャットの中にだけ成果物がある状態は、再利用性も監査性も低い。

Aionix では、Git リポジトリを主な Artifact Store として扱う。

Git 管理することで、以下が得られる。

- 差分が見える
- 履歴が残る
- レビューできる
- 戻せる
- 公開できる
- 他の AI / 人間が引き継げる

### 4.5.1 Kernel space と User space の分離

今回の Aionix では、OS 的な対応をさらに具体化している。

- `Kernel space` は `.github/`
- `Design space` は `design/`
- `User space` は `/root`

である。

`.github/` は、Aionix の control plane として扱う。ここには workflow、policy、approval rule、package admission rule、release gate のような、実行ルールを決めるものを置く。

`design/` は、Aionix を作るための設計書や実装メモを置く別空間である。ここは runtime そのものではなく、設計判断を蓄積し、人間と Agent が読み書きするための場所として使う。

`/root` は、Agent が実際に動作する user space である。workspace、runtime state、package 展開先、log、socket、lock などは原則としてこちら側に属する。

重要なのは、`.github/` が `統治する場所`、`design/` が `考える場所`、`/root` が `作業させる場所` だという点である。

さらに `/root` 以下は `LHF` に従って整理する。ここでは `LHF` を、Aionix user space を役割別に分割するための階層ルール名として扱う。

### 4.6 Permission = Tool Access / File Access / Approval Policy

AI に対して全権限を与えると危険である。

Aionix では、Agent ごとに権限を定義する。

たとえば、文書作成 Agent には Markdown 作成権限を与えてよいが、本番環境へのデプロイ権限を与えるべきではない。

権限は最低限にする。

高リスク操作には Human Approval を要求する。

### 4.7 Signal = Human interruption / Approval / Cancellation

OS における Signal は、プロセスへ割り込みや終了を伝える仕組みである。

Aionix では、人間からの割り込み、承認、停止、方針変更が Signal に相当する。

例:

- そのファイルは更新しないで
- その方向ではなく別案にして
- いったん保存して
- PR は作ってよいが merge はしないで
- 削除はしないで

AI 作業は長くなるほど、人間の途中介入が重要になる。

## 5. Agent と Skill の違い

Aionix では、Agent と Skill を混同しないことが最も重要である。

### 5.1 Agent とは何か

Agent は、目的を持って判断する主体である。

Agent は以下を行う。

- 目的を理解する
- 必要な情報を集める
- 実行計画を立てる
- 適切な Skill を選ぶ
- 実行結果を評価する
- 必要なら別の Skill を呼ぶ
- 成果物を保存する
- 人間に報告する

Agent は、作業の文脈を理解し、複数の選択肢から判断する。

### 5.2 Skill とは何か

Skill は、Agent が利用する機能単位である。

Skill は以下を行う。

- 入力を受け取る
- 定義された処理を実行する
- 出力を返す
- 失敗した場合はエラーを返す

Skill は、目的を自分で決めない。

### 5.3 配布単位としての Package

ここでさらに重要なのは、`導入単位` を Agent や Skill 単体にしないことである。

Aionix では、利用者が install する単位を `Aionix Package` とする。

Package は、少なくとも次をまとめて持つ。

- Agent
- その Agent が前提とする Skill 群
- manifest
- version
- dependency 情報
- optional templates / policies / assets

つまり、

- Agent は判断主体
- Skill は機能部品
- Package は導入単位

である。

### 5.4 例で理解する

たとえば、ユーザーが「Aionix の詳しい解説書を作ってアップして」と依頼したとする。

このときの流れは次のようになる。

1. Document Writer Agent が依頼を受け取る
2. File Governor Agent が保存先とファイル名を判断する
3. Markdown Skill が文書構造を整える
4. GitHub Skill がリポジトリにファイルを作成する
5. Observation Logger が出力先とコミットを記録する
6. Agent がユーザーに完了を報告する

ここで、判断しているのは Agent である。

GitHub へ保存する機能は Skill または Tool である。

利用者が環境へ入れるのは、たとえば `document-writer package` であり、その中に `Document Writer Agent`、`Markdown Skill`、`GitHub Skill`、必要なテンプレートや設定が含まれる。

## 6. Core Components 詳解

## 6.1 Task Manager

Task Manager は、ユーザー要求を実行可能なタスクとして扱う。

タスクには以下を持たせる。

```yaml
task_id: 20260424-001
request: Aionix の詳しい解説書を作成してアップする
status: running
agent: document-writer
required_skills:
  - markdown
  - github
artifacts:
  - design/003_202604242245_Aionix_詳解ガイド.md
approval_required: false
```

Task Manager の責務は、作業を単なる会話ではなく、追跡可能な単位にすることである。

## 6.2 Agent Runtime

Agent Runtime は、Agent を実行するための仕組みである。

主な責務:

- Agent 定義を読み込む
- Agent に Context を渡す
- Agent が Skill を呼べるようにする
- 実行結果を受け取る
- エラーを扱う
- ログを残す

初期実装では、Markdown で Agent 定義を管理し、Python CLI などで読み込む形が現実的である。

## 6.3 Context Manager

Context Manager は、Agent が判断に使う情報を整理する。

Context には以下が含まれる。

- ユーザーの依頼
- 直近の会話
- 対象リポジトリ
- 既存ファイル一覧
- 関連ドキュメント
- 作業制約
- 命名ルール
- 既存の設計思想

Context Manager の重要な役割は、Agent に何でも見せるのではなく、必要なものを適切に渡すことである。

## 6.4 Policy Engine

Policy Engine は、作業上のルールを適用する。

例:

- ルート直下に成果物を置かない
- 既存ファイル更新を優先する
- 新規ファイルには連番と日時を付ける
- 人間向け文書を `.agents/` 配下に置かない
- 削除操作は人間承認を必要とする
- 秘密情報を外部へ出さない

Policy Engine は Aionix における Kernel Policy に近い。

## 6.5 Permission Manager

Permission Manager は、Agent が何をしてよいかを管理する。

たとえば、以下のような権限を扱う。

```yaml
agent: document-writer
permissions:
  read_file: true
  write_file: true
  update_file: true
  delete_file: false
  create_pr: false
  merge_pr: false
  run_script: false
```

Permission Manager がないと、Agent の役割分離が崩れる。

## 6.6 Skill Dispatcher

Skill Dispatcher は、Agent からの Skill 呼び出しを受け付ける。

責務:

- Skill の存在確認
- 入力スキーマの確認
- 権限確認
- Skill 実行
- 出力の検証
- エラー処理

Skill Dispatcher を挟むことで、Agent が勝手に外部操作を行うことを防げる。

## 6.6.1 Package Manager

Package Manager は、Aionix Package を導入・更新・削除・固定するための中核コンポーネントである。

責務:

- Package manifest の読み込み
- package 名、version、dependency の解決
- Agent と bundled skills の展開
- install / update / remove / pin の管理
- 互換性チェック
- package ごとの required permissions の確認

ここで重要なのは、Package Manager が `Skill registry` より広い責務を持つことである。

Skill registry だけでは、どの Agent とどの Skill を一緒に導入すべきか、どの version 組み合わせが正しいかを保証できない。

そのため Aionix では、最終的に必要なのは `Aionix Package Manager` である。

### 6.6.2 なぜ Package Manager が必要か

Agent と Skill を別々に配布すると、次の問題が起きる。

- どの Skill がその Agent に必要か分からない
- version mismatch が起きる
- テンプレートやポリシーの同梱ができない
- 利用者が導入時に内部構造を理解しなければならない

Package 化すると、利用者は `何を入れれば使えるか` を package 名で理解できる。

### 6.6.3 典型的な package 例

- `aionix/document-writer`
- `aionix/reviewer`
- `aionix/research`
- `aionix/release-manager`

これらの package は、それぞれ Agent だけでなく、必要な Skill と最小設定を一緒に持つ。

### 6.6.4 最小 manifest 例

```yaml
name: aionix/document-writer
version: 0.1.0
agent: document-writer
skills:
  - markdown
  - github
  - file-governor
permissions:
  - read_file
  - write_file
  - update_file
dependencies: []
```

### 6.6.5 初期実装での package manager

初期段階では、Package Manager は複雑な外部 registry を持つ必要はない。

まずは次で十分である。

- ローカルディレクトリまたは Git repository から package を読む
- manifest を解釈する
- package を `.aionix/packages/` へ展開する
- 利用可能な Agent / Skill を runtime に登録する

つまり v1 では、`npm` や `apt` のような巨大な仕組みではなく、`manifest-aware installer` として始めるのが現実的である。

## 6.7 Artifact Manager

Artifact Manager は、生成物を管理する。

管理対象:

- Markdown
- Excel
- PDF
- ソースコード
- 設定ファイル
- 図表
- ログ

Artifact Manager は、成果物がどこに保存され、何から生成され、どのタスクに紐づくかを記録する。

## 6.8 Observation Logger

Observation Logger は、AI 作業の監査ログを残す。

記録する内容:

- task_id
- agent
- skill
- input
- output
- updated files
- commit sha
- errors
- approval

ログは、AI を信用するためではなく、AI の行動を後から検証できるようにするためにある。

## 6.9 Background Daemon の考え方

Aionix における `daemon` は、Linux の常駐プロセスに近いが、役割はもっと限定的に考える。

重要なのは、`daemon` を判断主体にしすぎないことである。

- Agent は目的を理解して判断する
- Skill は単機能を実行する
- Daemon は背景で状態を維持し、配送し、記録し、監視する

つまり Aionix の daemon は、`常駐 Agent` というより `background runtime service` に近い。

### 6.9.1 daemon を導入する理由

AI 実行基盤では、すべてをリクエスト駆動にすると不都合が出る。

たとえば次のような処理は、単発タスクではなく背景処理の方が自然である。

- 実行ログの集約
- Agent 間メッセージの配送
- タスク状態の監視
- タイムアウト検出
- リポジトリ状態の監視
- イベント発火

このため、Aionix には限定された daemon 群を持たせる価値がある。

### 6.9.2 Log Daemon

`Log Daemon` は、Observation Logger と密接に連携する背景サービスである。

責務:

- 実行中の Agent / Skill 呼び出しからイベントを受け取る
- JSONL などの監査ログへ追記する
- task_id、agent、artifact、error、approval の関連付けを維持する
- バッファリングや flush を管理する
- 将来的には log rotation や shipper 連携の入口になる

重要なのは、Log Daemon 自体が判断しないことである。

Log Daemon は、`何を記録するか` のルールは Policy Engine や Observation Logger から受け取り、ひたすら安全に保存する。

### 6.9.3 Messaging Daemon

`Messaging Daemon` は、Agent 間のメッセージ配送や状態通知を扱う。

ここでいう messaging は、Slack 通知のような外部通知だけではない。

Aionix の内部では、次のようなメッセージがある。

- `Research Agent` から `Writer Agent` への結果通知
- `Review Agent` から `Release Manager Agent` への差し戻し通知
- `Permission Manager` から `Task Manager` への承認待ち通知
- `Human approval required` のイベント通知

Messaging Daemon の責務:

- メッセージの enqueue / dequeue
- 宛先 Agent または subsystem への配送
- 少なくとも一回の配送保証、または重複許容の設計
- task_id / correlation_id の維持
- 再試行と失敗キューの管理

これにより、Aionix の IPC を `shared context だけ` に閉じず、イベント駆動の構造へ拡張できる。

### 6.9.4 State / Task Watcher Daemon

`Task Watcher Daemon` は、実行中タスクの状態監視を行う。

責務:

- 実行中タスクの heartbeat 確認
- timeout 検知
- orphan task 検知
- stuck 状態の検出
- 再実行候補または human review 必須状態への遷移

Task Manager がタスクの正本を持つとしても、背景監視は daemon 化した方が整理しやすい。

### 6.9.5 File / Repo Event Daemon

`File Event Daemon` または `Repo Event Daemon` は、workspace と Git 状態の変化を検知する。

責務:

- 対象ディレクトリの変更検知
- branch 変更や dirty state の検知
- 特定イベントで Hook を発火
- 競合しやすい更新への警告

ただし、この層は初期実装では必須ではない。

MVP ではポーリングや明示呼び出しでもよい。daemon 化するのは、並列 Agent 実行や自動化が増えた段階で十分である。

### 6.9.6 Notification Daemon

`Notification Daemon` は、人間への通知や外部通知を扱う。

責務:

- PR 作成完了通知
- 承認待ち通知
- タスク失敗通知
- 長時間実行ジョブの進捗通知
- 将来的な Slack / email / MCP 経由通知

これは Messaging Daemon と分けてもよいし、初期実装では Messaging Daemon の外部出力先として扱ってもよい。

### 6.9.7 初期実装で何を作るべきか

Aionix の初期実装で優先度が高いのは次の順である。

1. Log Daemon
   - 監査可能性に直結する
2. Messaging Daemon
   - Agent 間連携と承認待ち管理に効く
3. Task Watcher Daemon
   - 長時間タスクや失敗検知に効く

一方で、次は後回しでよい。

- File / Repo Event Daemon
- Notification Daemon の外部連携高度化

### 6.9.8 設計上の注意

daemon を増やしすぎると、Aionix はすぐに分散システムの複雑さを背負う。

そのため次を守る。

- daemon は判断主体にしない
- daemon は state machine と queue を明示する
- daemon 間通信は task_id を必ず持つ
- daemon の永続化単位を明確にする
- daemon が落ちても再開できるようにする
- 最初から多機能にせず、1 daemon 1責務を守る

### 6.9.9 Aionix における位置づけ

設計上は、daemon は `Aionix Core` の内部 subsystem として置くのが自然である。

たとえば次のように整理できる。

- Task Manager: 正本の状態管理
- Agent Runtime: 実行主体の起動
- Observation Logger: 記録ポリシー
- Log Daemon: 背景記録処理
- Messaging Daemon: IPC / event delivery
- Task Watcher Daemon: timeout / stuck detection

この整理により、Agent と Daemon の責務が混ざらない。

## 7. Claude Code / CLI との対応

Aionix は、Claude の機能をそのまま使うだけではなく、OS 的な役割へマッピングする。

### 7.1 Claude Code

Claude Code は、Aionix における Shell / Interface に相当する。

人間が Claude Code に依頼し、Claude Code がリポジトリ内のファイルを読んだり書いたりする。

Aionix の観点では、Claude Code 自体が OS なのではない。

Claude Code は、Aionix Runtime に対する操作インターフェースである。

### 7.2 Claude CLI

Claude CLI は、よりスクリプトや自動化に近い Shell として扱える。

将来的に Aionix CLI を作る場合、Claude CLI と組み合わせて以下を実現できる。

- task 実行
- agent 選択
- skill 呼び出し
- artifact 出力
- log 記録

### 7.3 MCP

MCP は、Aionix では Device Driver / External Subsystem に相当する。

MCP は、AI が外部システムへアクセスするための標準化された接続口である。

例:

- GitHub MCP
- Confluence MCP
- Slack MCP
- Google Sheets MCP
- AWS MCP
- Browser MCP

OS でデバイスドライバがハードウェア差分を吸収するように、MCP は外部システム差分を吸収する。

### 7.4 Claude Skills

Claude Skills は、Aionix の Skill Layer に対応する。

ただし、Aionix では Skill をより厳密に扱う。

Skill には以下を明記する。

- 目的
- 入力
- 出力
- 副作用
- 失敗時の挙動
- 呼び出し可能な Agent
- バージョン

### 7.5 Hooks

Hooks は、Aionix では Signal Handler / Event Hook に相当する。

例:

- ファイル保存前に File Governor を呼ぶ
- PR 作成前に Review Agent を呼ぶ
- コマンド実行前に Permission Manager を呼ぶ
- 成果物生成後に Observation Logger を呼ぶ

Hooks は強力だが、濫用すると動作が分かりにくくなる。

そのため、Aionix では Hooks の役割を限定し、重要なイベントだけに適用する。

## 9. Agent 定義の書き方

Agent は Markdown で定義できる。

例:

```markdown
# Document Writer Agent

## Role

人間向け文書を作成・更新する Agent。

## Responsibilities

- ユーザー依頼を文書構成へ変換する
- 既存文書を尊重して更新する
- Markdown として読みやすく整える
- 保存前に File Governor の判断を使う

## Permissions

- read_file
- write_file
- update_file

## Forbidden

- delete_file
- merge_pr
- access_secret

## Skills

- markdown
- github
- file-governor
```

Agent 定義では、何をしてよいかだけでなく、何をしてはいけないかを明記する。

AI は制約が曖昧だと、良かれと思って余計なことをする。

そのため、禁止事項は明示的に書く。

## 10. Skill 定義の書き方

Skill も Markdown で定義できる。

例:

```markdown
# Markdown Formatter Skill

## Purpose

Markdown 文書を読みやすく整形する。

## Input

- raw_markdown: string
- style: optional string

## Output

- formatted_markdown: string

## Side Effects

なし。

## Failure

入力が空の場合は error を返す。

## Version

0.1.0
```

Skill では、副作用の有無を必ず書く。

副作用がない Skill は安全に再利用しやすい。

副作用がある Skill は権限管理が必要になる。

## 11. File Governor の重要性

Aionix において、File Governor は非常に重要である。

AI に作業を任せると、似たファイルが増えやすい。

例:

- `設計書.md`
- `設計書_v2.md`
- `設計書_修正版.md`
- `設計書_最新版.md`
- `new_design.md`

この状態になると、正本が分からなくなる。

File Governor は、以下を判断する。

- 既存ファイルを更新すべきか
- 新規ファイルを作るべきか
- 派生ファイルとして作るべきか
- 保存先はどこか
- ファイル名はどうするか
- ルート直下に置いていないか

AI 作業では、コンテンツ生成能力よりも、成果物管理能力の方が後から効いてくる。

## 12. fcntl によるロック制御

Aionix では、ファイルやタスクの同時更新を防ぐためにロック制御を設計する。

古い `lockf` 前提ではなく、POSIX 互換性を意識して `fcntl` ベースのロックを採用する。

### 12.1 なぜロックが必要か

AI エージェントが複数動くと、同じファイルを同時に更新する可能性がある。

たとえば、Document Writer Agent と Review Agent が同じ Markdown を同時に編集すると、片方の変更が失われる可能性がある。

これを防ぐために、更新対象ごとにロックを取る。

### 12.2 ロック対象

- ファイル単位
- タスク単位
- リポジトリ操作単位
- PR 作成単位

### 12.3 ロックファイルの配置

```text
.aionix/locks/
├── file_Docs_documents_003.lock
├── task_20260424_001.lock
└── repo_main_write.lock
```

### 12.4 Python 実装例

```python
import fcntl
from pathlib import Path

class FileLock:
    def __init__(self, lock_path: str):
        self.lock_path = Path(lock_path)
        self.lock_path.parent.mkdir(parents=True, exist_ok=True)
        self.file = None

    def __enter__(self):
        self.file = self.lock_path.open("w")
        fcntl.flock(self.file.fileno(), fcntl.LOCK_EX)
        return self

    def __exit__(self, exc_type, exc, tb):
        if self.file:
            fcntl.flock(self.file.fileno(), fcntl.LOCK_UN)
            self.file.close()

with FileLock(".aionix/locks/example.lock"):
    # safe write operation
    pass
```

この実装は Linux / macOS を主対象にする。

Windows 対応が必要な場合は、別実装または抽象化レイヤーを用意する。

## 12.5 Ubuntu ユーザランド起動を Aionix に実装すべきか

ここでいう `Linux が起動するユーザランドプロセス` とは、単に Linux カーネル上でコマンドを実行することではなく、特定ディストリビューションの root filesystem、初期プロセス、ライブラリ、パッケージ構成を持つ実行環境を Aionix 自身がどこまで内包するか、という設計論点である。

`2026-04-24` 時点の前提として、最新の Ubuntu は `Ubuntu 26.04 LTS (Resolute Raccoon)` である。  
この前提で考えると、Aionix の core に Ubuntu 起動機構そのものを入れるのは、初期実装としては採用しない方がよい。

### 理由 1. Aionix の責務が OS 互換層へ膨張する

Aionix の本来の中心は、Agent、Skill、Context、Permission、Artifact、Log を管理することである。

ここに Ubuntu userland の起動責務まで持ち込むと、次のような問題が一気に増える。

- rootfs の配布と更新
- init をどう扱うか
- `systemd` 前提の挙動をどこまで再現するか
- `/proc`、`/sys`、`/dev`、`tmpfs` の扱い
- namespace / cgroup / mount 設計
- ネットワーク分離
- パッケージ更新責任
- セキュリティパッチ追従

これは AI 実行基盤の実装というより、コンテナランタイムまたは軽量 VM 基盤に近い問題になる。

### 理由 2. 再現性は userland 起動そのものより、実行契約で作れる

Aionix が必要としている再現性は、必ずしも `Ubuntu を boot すること` ではない。

むしろ重要なのは、次の契約を明示することである。

- どの OS 系列を対象にするか
- どの Python / shell / package manager を前提にするか
- どのツール群が存在するか
- どの権限で動くか
- どのファイルシステム階層を使うか

この契約が明確であれば、初期段階では host OS 上でも十分に Aionix の価値を作れる。

### 理由 3. Ubuntu 完全起動を入れると MVP が遅くなる

Ubuntu 26.04 LTS をターゲットにするとしても、最初から `PID 1`、`systemd`、完全なディストリビューション起動を扱うと、MVP の焦点がぼやける。

初期段階で優先すべきなのは、次のような部分である。

- Agent 実行モデル
- Skill 呼び出し契約
- ファイル保存規律
- ロック
- ログ
- 権限と承認
- Git 連携

これらが固まる前に Ubuntu userland 起動へ入ると、Aionix の価値よりランタイム実装の重さが前面に出る。

### 結論

結論として、`Ubuntu 26.04 LTS を boot する userland 実装を Aionix core に直接入れるべきではない。`

ただし、`Ubuntu 26.04 LTS 互換の実行ターゲットを後から差し替え可能にする設計` は持つべきである。

つまり判断は次の通りである。

- `今すぐ core に実装する`: しない
- `将来の runtime target として設計対象にする`: する
- `最新 Ubuntu を互換ターゲットとして明示する`: する

### 採るべき実装方針

初期実装では、Aionix は `host OS 上で動く実行基盤` として作る。

そのうえで runtime abstraction を導入し、将来的に次の3段階を選べるようにする。

1. Host Runtime
   - Linux / macOS のホスト上で直接動かす
   - 最速で MVP を作れる

2. Ubuntu Rootfs Runtime
   - `Ubuntu 26.04 LTS` の rootfs を対象にした隔離実行
   - ただし `full boot` ではなく、`systemd を前提にしないプロセス実行` を基本にする

3. VM / MicroVM Runtime
   - より強い隔離や再現性が必要になった段階で追加する
   - これは v1 ではなく、後期段階の課題とする

### Aionix で本当に必要な最小要件

もし Ubuntu ターゲットを扱うなら、最初に必要なのは `Ubuntu を起動すること` ではなく、次の最小要件である。

- `Ubuntu 26.04 LTS` 互換の filesystem layout
- `/bin/sh` または `bash` の存在
- Python 実行環境
- 必須ツール群の存在確認
- ワークスペースの mount
- 実行ログの採取
- 権限制御

このレベルであれば、Aionix の主題を壊さずに Ubuntu 対応を進められる。

### 設計原則として残すべき一文

Aionix は Ubuntu を置き換えるものでも、Ubuntu を起動すること自体を主目的とするものでもない。  
Aionix は、必要に応じて Ubuntu 26.04 LTS 互換の runtime target を利用できる AI 実行基盤として設計する。

## 13. 実行ログ設計

AI の行動は、後から追跡できなければならない。

ログには、思考の全文ではなく、監査に必要な要約を残す。

例:

```json
{
  "task_id": "20260424-003",
  "request": "Aionix の詳しい解説書を作成してアップする",
  "agent": "document-writer",
  "skills": ["markdown", "github", "file-governor"],
  "artifacts": [
    "design/003_202604242245_Aionix_詳解ガイド.md"
  ],
  "repository": "eyes-919/aionix",
  "branch": "main",
  "status": "completed"
}
```

重要なのは、AI の全思考を残すことではなく、次の事実を追えるようにすることである。

- 何を頼まれたか
- 何をしたか
- どのファイルを作ったか
- どのルールを使ったか
- 成功したか失敗したか

## 14. Human Approval の設計

Aionix では、人間承認をどこで必要にするかを明確にする。

### 14.1 承認不要にできる操作

- 新規 Markdown の作成
- 既存文書への軽微な追記
- 一時ファイルの生成
- 読み取り専用の検索
- ログ記録

### 14.2 承認が必要な操作

- ファイル削除
- 大規模上書き
- 秘密情報へのアクセス
- 外部公開
- PR マージ
- 本番環境操作
- 課金が発生する操作

### 14.3 承認モデル

```yaml
operation: delete_file
risk: high
approval_required: true
approver: human
```

AI の安全性は、AI の人格に依存させてはいけない。

構造として、人間承認が必要な場所を固定する。

## 17. よくある設計ミス

### 17.1 Agent に何でもやらせる

Agent に調査、判断、保存、削除、公開、レビューを全部やらせると危険である。

Agent は役割ごとに分ける。

### 17.2 Skill を Agent 化する

Skill は判断主体ではない。

Skill に自律判断を入れ始めると、責務が曖昧になる。

### 17.3 ファイル保存ルールを後回しにする

AI 開発では、最初にファイルが散らかる。

保存ルールは最初に作る。

### 17.4 ログを残さない

AI が何をしたか分からない状態は、運用に乗らない。

最低限、入力、出力、更新ファイル、実行 Agent は残す。

### 17.5 Human Approval を曖昧にする

AI の安全性は、AI の善意ではなく、承認設計で担保する。

## 18. 設計判断の原則

Aionix を拡張するときは、以下の原則に従う。

1. 判断主体は Agent に置く
2. 再利用機能は Skill に置く
3. 副作用は明示する
4. 保存先は File Governor が決める
5. 高リスク操作は人間承認にする
6. 実行ログを残す
7. Git で差分管理できる形にする
8. 公開可能な情報と非公開情報を混ぜない
9. 最初から巨大な自律システムにしない
10. 小さく動くものを積み上げる

## 19. Aionix の価値

Aionix の価値は、AI を賢く使うことそのものではない。

価値は、AI の作業を構造化し、人間が主導権を保ちながら、継続的に成果物を積み上げられることにある。

AI は今後さらに高機能になる。

しかし、高機能になるほど、次の問題が強まる。

- 何を任せてよいか
- 何を任せてはいけないか
- 何を根拠に判断したか
- 成果物はどこにあるか
- どれが正本か
- 誰が承認したか
- 失敗時にどう戻すか

Aionix は、この問題に対する構造的な答えである。

## 20. まとめ

Aionix は、AI エージェント時代の作業基盤を OS 的に再設計する試みである。

中心にある考え方は明確である。

- Agent は判断主体
- Skill はライブラリ
- Context は実行時情報
- Memory は長期知識
- Artifact は Git 管理される成果物
- Permission は最小権限
- Log は監査のために残す
- Human Approval は高リスク操作の境界に置く

これらを分離して扱うことで、AI 活用は単なるプロンプト芸から、再現可能な実行基盤へ進化する。

Aionix は、そのための名前であり、設計思想であり、今後実装していくべき公開可能なフレームワークである。
