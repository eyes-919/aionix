# Aionix 全体設計

作成日: 2026-04-24

## 1. 目的

Aionix は、Claude Code / CLI などの AI 実行環境を、単なるプロンプト実行ツールではなく、OS 的な構造で扱うための設計思想および実装基盤である。

目的は、AI エージェント、スキル、メモリ、ファイル、プロセス、権限、実行制御、観測、ログ、リポジトリ運用を一つの体系として整理し、人間と AI が継続的に共同作業できる作業環境を作ることである。

Aionix は、AI そのものを OS にするのではない。AI エージェントが安全かつ再現可能に動作するための、OS ライクな実行基盤を定義する。

## 2. 基本思想

### 2.1 AI OS ではなく AI 実行基盤

Aionix は、Linux のような汎用 OS を置き換えるものではない。

むしろ、既存 OS の上に構築される AI ネイティブな実行レイヤーである。

Linux がプロセス、ファイル、権限、デバイス、ネットワークを抽象化したように、Aionix は以下を抽象化する。

- エージェント
- スキル
- コンテキスト
- メモリ
- タスク
- ファイル操作
- ツール呼び出し
- 実行権限
- 成果物
- 観測ログ
- レビュー
- 人間による承認

### 2.2 エージェントとスキルの分離

Aionix における重要な設計原則は、エージェントとスキルを明確に分離することである。

- エージェントは判断主体である
- スキルは再利用可能な機能ライブラリである

スキルは自律的に意思決定しない。複数のエージェントから呼び出される共通ライブラリであり、特定の処理、変換、生成、検証、操作を担当する。

一方、エージェントは目的を理解し、スキルを選択し、実行順序を決め、成果物を評価する。

この分離により、Aionix はスキルの再利用性とエージェントの専門性を両立させる。

## 3. 用語定義

### 3.1 Agent

Agent は、目的を持って行動する実行主体である。

Agent は以下を持つ。

- Role
- Goal
- Policy
- Memory access scope
- Tool access scope
- Skill access scope
- Output contract
- Review responsibility

例:

- File Governor Agent
- Document Writer Agent
- Code Reviewer Agent
- Research Agent
- Release Manager Agent
- Cost Observer Agent

### 3.2 Skill

Skill は、Agent が呼び出す再利用可能な処理単位である。

Skill はライブラリであり、Agent ではない。

例:

- Markdown Formatter Skill
- GitHub Repository Skill
- Spreadsheet Generation Skill
- AWS Cost Estimation Skill
- Security Review Skill
- PDF Export Skill
- Diagram Generation Skill

Skill は以下を持つ。

- Input schema
- Output schema
- Preconditions
- Side effects
- Error contract
- Version

### 3.2.1 Aionix Package

Aionix では、Agent や Skill を単独で導入するのではなく、`Agent` と、その Agent が前提とする `Skill` 群をひとまとめにした `Aionix Package` を導入単位とする。

考え方は次の通りである。

- 実行主体: Agent
- 再利用機能: Skill
- 配布 / 導入単位: Aionix Package

Package は少なくとも次を持つ。

- Agent definition
- Bundled skills
- Package manifest
- Version
- Dependency metadata
- Required permissions
- Optional templates / policies / assets

### 3.3 Context

Context は、Agent が判断に使用する現在の作業情報である。

Context には以下が含まれる。

- ユーザー要求
- 現在の会話
- 対象リポジトリ
- 参照ドキュメント
- 作業中ファイル
- 制約条件
- 実行履歴

### 3.4 Memory

Memory は、Context より長期に保持される知識である。

Aionix では Memory を以下に分ける。

- User memory
- Project memory
- Repository memory
- Agent memory
- Runtime memory

Memory は無制限に参照してよいものではなく、Agent ごとにアクセススコープを定義する。

### 3.5 Task

Task は、Agent が実行する作業単位である。

Task は以下を持つ。

- Request
- Plan
- Inputs
- Expected outputs
- Required skills
- Permissions
- Status
- Logs
- Result

### 3.6 Artifact

Artifact は、Agent が生成または更新する成果物である。

例:

- Markdown
- Excel
- Python script
- GitHub Pull Request
- Configuration file
- Design document
- Diagram

## 4. OS 概念とのマッピング

Aionix は、従来 OS の概念を AI 実行環境へ対応づけて設計する。

| OS 概念 | Aionix における対応 |
|---|---|
| Process | Agent 実行インスタンス |
| Thread | Agent 内のサブタスク |
| System call | Tool / Skill 呼び出し |
| Library | Skill |
| File system | Workspace / Repository / Artifact store |
| Shell | Claude Code / CLI / Chat interface |
| User | Human operator |
| Permission | Tool access / File access / Approval policy |
| Scheduler | Task orchestrator |
| Kernel space | `.github/` に置く control plane / governance plane |
| User space | `/root` に置く agent runtime / workspace / state |
| Kernel policy | Core governance rules |
| Logs | Execution trace / Decision log |
| Package manager | Aionix package manager |
| Daemon | Background runtime service |
| IPC | Agent-to-agent message / event delivery / shared context |
| Lock | File / Task / Resource lock |
| Signal | Human interruption / cancellation / approval |
| Exit code | Task result status |

## 6. レイヤー設計

### 6.1 Interface Layer

Interface Layer は、人間と Aionix の接点である。

想定インターフェース:

- ChatGPT
- Claude Code
- Claude CLI
- VS Code
- GitHub Issue / PR
- Slack
- Local shell

この層は入力を受け取るだけでなく、成果物の提示、差分確認、承認依頼も担当する。

### 6.2 Aionix Core

Aionix Core は、中核制御層である。

主な責務:

- タスクの受理
- エージェント選択
- スキル呼び出し
- Package の解決と導入
- 権限確認
- 作業ディレクトリ決定
- ファイル保存方針の適用
- 実行ログの記録
- 人間承認の要求
- エラー処理
- 背景 daemon の管理

補足:

- `Log Daemon` は監査ログの背景保存を担当する
- `Messaging Daemon` は Agent 間メッセージやイベント配送を担当する
- `Task Watcher Daemon` は timeout や stuck task の監視を担当する
- `Package Manager` は Aionix Package の導入、更新、依存解決、バージョン固定を担当する

これらの daemon は判断主体ではなく、Aionix Core の background subsystem として扱う。

### 6.3 Agent Layer

Agent Layer は、目的別の判断主体を提供する。

Agent は専門性を持つが、共通ルールに従う。

例:

#### File Governor Agent

- 保存場所を判断する
- 既存ファイル更新か新規作成かを判断する
- 命名規則を適用する
- ルート直下への成果物保存を防ぐ

#### Document Writer Agent

- 文書構成を設計する
- Markdown を生成する
- 既存文書の文体を維持して更新する

#### Code Reviewer Agent

- 差分を読む
- リスクを分類する
- 修正提案を出す
- レビューコメントを作る

#### Release Manager Agent

- ブランチ戦略を確認する
- リリース条件を検査する
- PR 運用ルールと照合する

### 6.4 Skill Layer

Skill Layer は、Agent が共通利用するライブラリ群である。

Skill はできるだけ純粋関数に近づける。

ただし、GitHub 操作やファイル作成のような副作用を持つ Skill も存在する。その場合は、入力、出力、副作用、失敗条件を明確にする。

### 6.4.1 Package Layer の考え方

内部構造として Agent Layer と Skill Layer は分けるが、利用者に見せる導入単位は Package Layer とする。

つまり、

- Skill は内部部品
- Agent は判断主体
- Package は install / update / remove の単位

である。

### 6.5 Workspace / Repository Layer

Workspace は、Aionix が作業する実体である。

原則:

- すべての成果物は Git 管理できる形で保存する
- 生成物と生成元の関係を明示する
- 人間向け文書と Agent 用文書を分離する
- ルート直下は管理ファイルに限定する

### 6.5.1 `.github` と `/root` の役割分担

今回の Aionix では、OS 的な比喩をさらに進めて、制御面、設計面、作業面を明確に分ける。

- `Kernel space` は `.github/` として扱う
- `Design space` は `design/` として扱う
- `User space` は `/root` として扱う

`.github/` は、Aionix の control plane である。ここには、workflow、policy、approval gate、package admission、release rule のような、`何を許し、どう流すか` を決めるものを置く。

`design/` は、Aionix 自体を作るための設計書、アーキテクチャ、判断メモ、実装計画を置く別空間である。ここは runtime ではなく、人間と Agent が設計を整理するための領域として扱う。

一方で `/root` は、Agent が実際に動き、workspace を持ち、state を持ち、artifact を生成する作業空間である。

さらに `/root` 以下は、Aionix が採用する `LHF` に従って整理する。ここでいう `LHF` は、Aionix の user space を乱雑にせず、役割別に分けるための階層ルール名として扱う。


この `/root` 以下の階層は、LHF に従って固定する。

## 8. 実装方針

### 8.1 最初に作るもの

初期実装では、巨大なフレームワークを作らない。

まず GitHub に公開できる最小構成を作る。

優先順:

1. README.md
2. `.github/` 配下の kernel-space 用 workflow / policy / release rule
3. `design/` 配下の設計文書
4. `runtime/core/` と `runtime/daemons/` の骨格
5. `runtime/manifests/` 配下の package manifest
6. `agents/` と `skills/` の最小定義
7. サンプル task / package

### 8.2 実行可能性

Aionix は、思想だけでなく実際に動くものとして公開する。

最初の実装では、以下を対象とする。

- Markdown ファイルの作成
- 既存ファイル更新
- GitHub リポジトリへの保存
- Agent / Skill 定義の読み込み
- タスク実行ログの保存

OS / runtime 方針:

- Aionix core は host OS 上で動く実行基盤として開始する
- Linux 起動機構や Ubuntu boot 機構そのものは core に含めない
- 将来的な runtime target として `Ubuntu 26.04 LTS` 互換環境を設計対象にする

### 8.3 ロック制御

古い `lockf` 前提ではなく、POSIX 互換性を意識しつつ `fcntl` ベースのファイルロックを採用する。

目的:

- 同一ファイルへの同時書き込み防止
- タスク単位の排他制御
- Artifact 更新時の競合回避

設計方針:

- ロック対象はファイルまたはタスク ID
- ロックファイルは `.aionix/locks/` に置く
- ロック取得に失敗した場合は待機、再試行、失敗を明示する
- 強制解除は人間承認を必要とする

### 8.4 状態管理

初期段階では、状態管理はファイルベースでよい。

将来的には SQLite に移行できる設計にする。

候補:

```text
.aionix/
├── runtime/
├── logs/
├── locks/
├── tasks/
└── state/
```

ただし、公開リポジトリでは `.aionix/runtime/` や `.aionix/logs/` は `.gitignore` 対象にする。

## 9. 権限モデル

Aionix では、Agent ごとに権限を分ける。

### 9.1 権限種別

- read_file
- write_file
- update_file
- delete_file
- create_branch
- create_pr
- merge_pr
- call_external_tool
- access_memory
- access_secret
- run_script

### 9.2 原則

- デフォルトは最小権限
- 削除、上書き、外部公開、PR マージは高リスク操作とする
- 高リスク操作は人間承認を要求する
- Agent は自分の権限を拡張できない
- Skill は Agent の権限を超えて実行できない

## 10. 観測性

Aionix は、AI が何をしたかを後から追えるようにする。

記録するもの:

- 入力要求
- 選択された Agent
- 呼び出された Skill
- 参照したファイル
- 更新したファイル
- 判断理由の要約
- エラー
- 人間承認の有無
- 生成物パス

ログ形式例:

```json
{
  "task_id": "20260424-001",
  "agent": "document-writer",
  "skills": ["markdown", "github"],
  "inputs": ["user_request"],
  "outputs": ["design/002_202604242230_Aionix_全体設計.md"],
  "status": "completed"
}
```

## 11. Claude 最新機能との対応

Claude Code / CLI を前提にすると、Aionix は次のように対応する。

| Claude 側の概念 | Aionix での扱い |
|---|---|
| Claude Code | Interface / Shell |
| Claude CLI | Shell / Runtime frontend |
| Tool use | System call |
| MCP | Device driver / External subsystem |
| Skills | Library / Package |
| Agents | Process / Service |
| Project instructions | Policy / Kernel config |
| Memory | Long-term memory layer |
| Hooks | Event hook / Signal handler |
| GitHub integration | Repository device / VCS subsystem |

重要なのは、Claude の機能をそのまま使うのではなく、OS 的な構造にマッピングして管理可能にすることである。

## 12. セキュリティとガバナンス

Aionix は、AI に善意を仮定しない。

AI は便利な実行主体だが、常に誤る可能性がある。そのため、以下を必須とする。

- 権限分離
- 実行ログ
- 人間承認
- 差分確認
- 削除操作の制限
- 外部送信の制御
- 秘密情報へのアクセス制限
- 生成物のレビュー

## 13. 最小実装 MVP

MVP では、以下を実現する。

### 13.1 機能

- Agent 定義を Markdown で管理する
- Skill 定義を Markdown で管理する
- Package manifest を Markdown または YAML で管理する
- ファイル保存ルールを File Governor Agent が判断する
- ドキュメントを生成して GitHub に保存する
- 実行ログを Markdown または JSON で残す

### 13.2 非対象

初期段階では以下は対象外とする。

- 完全な自律実行
- 常駐デーモン
- 複雑なスケジューラ
- Agent 間の高度な並列実行
- 本格的な権限サンドボックス
- 完全な package registry federation

## 14. 公開リポジトリ化に向けた方針

Aionix は、GitHub で公開できる形を目指す。

公開時に必要なもの:

- README
- ライセンス
- 基本思想
- Agent / Skill のサンプル
- ディレクトリ構成
- 実行例
- ロードマップ
- セキュリティ方針

ただし、個人情報、社内情報、実プロジェクト固有情報は含めない。

## 16. 結論

Aionix は、AI を単なる作業補助として扱うのではなく、AI エージェントが安全に、再現可能に、継続的に成果物を生み出すための実行基盤として設計する。

重要なのは、エージェント、スキル、コンテキスト、メモリ、権限、成果物、観測性を分離して扱うことである。

特に、スキルをライブラリとして扱い、エージェントを判断主体として扱い、それらを Package として導入可能にする設計は、今後の AI 開発環境を整理する上で中核になる。

Aionix は、Claude の最新機能を含む AI 実行環境を OS 的に再解釈し、人間が主導権を保ちながら AI を使うための設計基盤である。
