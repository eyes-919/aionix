# Ubuntu 最小機能一覧と実装リスト

作成日: 2026-04-24

## 1. この文書の目的

この文書は、`AIONIX が Ubuntu 互換 runtime target を将来的に扱う` 前提で、Ubuntu が公式に提供している機能領域を最小単位で一覧化し、AIONIX 側で必要になる実装項目へ落とし込むための作業メモである。

ここでは、まず `Ubuntu が公式情報として提供している機能領域` を整理し、その後で `AIONIX 実装リスト` を作る。

## 2. 前提

- `2026-04-24` 時点の最新 Ubuntu LTS は `Ubuntu 26.04 LTS (Resolute Raccoon)`
- この文書の `Ubuntu 側の機能一覧` は、主に Ubuntu / Canonical の公式ドキュメントと公式アナウンスを参照している
- この文書の `AIONIX 実装リスト` は、上記の公式情報をもとにした設計整理であり、Ubuntu 公式仕様そのものではない

## 3. Ubuntu が最低限提供していると見なせる機能領域

### 3.1 リリースとサポートの提供

Ubuntu は、リリースサイクル、サポート期間、セキュリティメンテナンス期間を明示して提供している。

最低限の機能として見なせる要素:

- リリース番号とコードネーム
- 対応アーキテクチャ
- 固定リリースとしての提供
- サポート期間の定義
- セキュリティ更新提供期間の定義

### 3.2 インストーラと自動インストール

Ubuntu Server 公式ドキュメントでは、通常インストールに加えて、自動インストールやアーキテクチャ別の導入方法を提供している。

最低限の機能として見なせる要素:

- インストーラ
- 自動インストール機構
- 対応アーキテクチャごとの導入経路
- ISO / image 配布

### 3.3 パッケージ管理とアーカイブ

Ubuntu は、APT / dpkg とリポジトリを前提に、ソフトウェアの導入、更新、削除、アップグレードを提供している。

最低限の機能として見なせる要素:

- パッケージフォーマット
- パッケージマネージャ
- リポジトリ設定
- パッケージインデックス更新
- パッケージ導入 / 削除 / 更新
- ログ記録

### 3.4 セキュリティ更新

Ubuntu は、サポート対象リリースに対して、コンポーネントごとのセキュリティ更新提供方針を持っている。

最低限の機能として見なせる要素:

- セキュリティ更新経路
- サポート対象コンポーネントの区分
- 互換性を重視した更新方針
- 自動セキュリティ更新

### 3.5 ユーザーと権限管理

Ubuntu は、ローカルユーザー管理と `sudo` ベースの権限昇格モデルを提供している。

最低限の機能として見なせる要素:

- ユーザー作成 / 削除
- グループ管理
- root 直接利用を前提にしない管理者権限モデル
- `sudo` による権限昇格

### 3.6 ネットワーク設定

Ubuntu Server では、Netplan を用いたネットワーク設定を公式の中心に置いている。

最低限の機能として見なせる要素:

- インターフェース設定
- IP 設定
- DNS / name resolution の基礎設定
- bridge を含む基本ネットワーク抽象

### 3.7 時刻同期

Ubuntu は、時刻同期のための仕組みを提供している。  
`Ubuntu 25.10` 以降は `chrony` がデフォルトで、`timedatectl` / `timesyncd` は補助的に扱われる。

最低限の機能として見なせる要素:

- 時刻同期サービス
- timezone 設定
- status 確認
- NTP 設定

### 3.8 ログ基盤

Ubuntu の周辺ドキュメントでは、`journald` や `/var/log` を使ったログ参照が前提になっている。

最低限の機能として見なせる要素:

- システムログ保存
- サービスログ参照
- パッケージ操作ログ
- 構造化されたログ参照の基盤

### 3.9 サービス / init 基盤

Ubuntu のドキュメントでは、`systemd` の unit file、`systemctl`、`timedatectl`、`journald` などが前提になっている。

最低限の機能として見なせる要素:

- init / service manager
- サービス起動 / 停止 / 状態確認
- unit file の配置
- boot 後の常駐サービス管理

### 3.10 初期化 / プロビジョニング

Ubuntu は、cloud image や cloud-init を通じて、初回起動時の初期化を提供している。

最低限の機能として見なせる要素:

- 初回起動時の初期設定
- hostname / user / network の初期化
- スクリプト実行
- repeatable な provisioning

### 3.11 セキュリティ基盤

Ubuntu では、AppArmor がデフォルトで入っており、Mandatory Access Control を補完している。

最低限の機能として見なせる要素:

- 基本的な MAC 機構
- プロファイル適用
- profile reload
- deny ログ確認

### 3.12 ストレージ基盤

Ubuntu Server は、最低限のストレージ要件、インストール先、LVM などの基礎的なストレージ構成を前提にしている。

最低限の機能として見なせる要素:

- root filesystem
- 永続ストレージ配置
- インストール先の初期化
- 必要に応じた LVM 等の基礎構成

## 4. AIONIX で最低限実装すべき項目

以下は `Ubuntu のすべてを再実装する` ための一覧ではない。  
`AIONIX が Ubuntu 26.04 LTS 互換 runtime target を扱うために、最低限どこまで持つべきか` という実装リストである。

### 4.1 P0 必須

- Ubuntu 26.04 LTS を target version として明示する
- 対応アーキテクチャを明示する
- rootfs / image の取得元と version pin の仕組みを持つ
- package manager の前提を `apt` / `dpkg` ベースで固定する
- repository 定義と署名鍵の扱いを定義する
- package install / update / remove を実行できる
- security update の適用経路を持つ
- user / group / sudo モデルを定義する
- Netplan 相当のネットワーク設定注入手段を持つ
- service manager 前提を `systemd` 系で固定する
- journald / パッケージログ / AIONIX ログの採取方針を持つ
- cloud-init 相当の first-boot provisioning 手段を持つ
- AppArmor を使うか、非対応として明示する
- workspace mount と artifact 保存先の標準位置を決める
- kernel space を `.github/`、user space を `/root` として分離する
- `/root` 以下の階層を LHF に従って固定する

### 4.2 P1 早期対応

- autoinstall 相当の non-interactive install フロー
- time sync の標準設定
- image build pipeline
- update channel の分離
- package snapshot / pinning
- basic health check
- rollback 前提の image / rootfs 管理

### 4.3 P2 後期対応

- 複数アーキテクチャ対応拡張
- ミラーや独自 archive の運用
- ESM 相当の長期更新戦略
- AppArmor profile の同梱 / 生成
- observability stack 連携
- VM / microVM runtime との統合

## 6. まず決めるべき論点

実装を始める前に、次の点は先に決めた方がよい。

- AIONIX は `Ubuntu full distribution runtime` を目指すのか
- それとも `Ubuntu-compatible execution target` を目指すのか
- `cloud-init` をそのまま使うのか
- `systemd` を必須前提にするのか
- host 上の隔離実行で始めるのか
- microVM まで初期から入れるのか
